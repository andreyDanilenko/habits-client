## PLAN_ROLE_STAGE_1_INHERITANCE — Наследование ролей (Этап 1)

### Цель этапа

Добавить полноценную поддержку наследования ролей на фронтенде: выбор родительской роли при создании/редактировании, визуализация унаследованных прав и интеграция с уже готовыми API наследования.

---

### 1. Изменения в UI формы роли (`RoleFormModal.vue`)

- **Добавить выбор родительской роли:**
  - В блоке формы добавить секцию:
    - Заголовок: “Наследование от роли”.
    - `select` со списком доступных ролей-родителей:
      - первая опция — “Не наследует” (значение `null`/`''`),
      - далее — список ролей, доступных для наследования (только кастомные, не системные).
  - Источник данных:
    - использовать роли из `useRolesPage()` (передавать список в `RoleFormModal` или запрашивать через отдельный хук/проп).

- **Поведение:**
  - При открытии модалки на редактирование роли:
    - выбранное в селекте значение должно отражать текущую конфигурацию наследования (родительская роль, если есть).
  - При создании роли:
    - по умолчанию — “Не наследует”.

---

### 2. Логика наследования в модели (`use-role-editor.ts`)

- **Расширить `RoleEditorOptions`:**
  - добавить опциональное поле `inheritsRoleId?: string | null` (или аналогичное).

- **Добавить состояние наследования:**
  - `const inherits = ref<string | null>(options.inheritsRoleId ?? null)`.

- **Унаследованные права:**
  - Завести `computed`:
    - `inheritedPermissions = computed<PermissionString[]>(...)`, которое:
      - если `inherits` пустое — возвращает `[]`,
      - иначе ищет выбранную родительскую роль в списке ролей (`useRolesPage().roles`) и возвращает её права (после загрузки).

- **Интеграция с `PermissionTree`:**
  - Расширить `PermissionTree.vue`, чтобы он мог:
    - отображать унаследованные права отличным стилем (например, чекбокс disabled + подпись “унаследовано”),
    - при этом локальные права роли (`permissions`) остаются редактируемыми.
  - Пропсы:
    - кроме `modelValue: string[]` добавить, например, `inherited: string[]`.

- **Сохранение:**
  - В методе `save()`:
    - сформировать payload для `createRole/updateRole` (как сейчас),
    - дополнительно вернуть/передать в вызывающий код значение `inherits` (родительская роль).

---

### 3. Интеграция с API наследования (`roleService`)

- **При сохранении роли:**
  - После успешного `createRole/updateRole`:
    - если выбрана родительская роль (есть `inherits`) и в БД ещё нет такой связи:
      - вызвать `roleService.addInheritance(workspaceId, childRoleId, parentRoleId)`.
    - если ранее была связь, но пользователь убрал наследование:
      - вызвать `roleService.removeInheritance(...)`.

- **При загрузке роли на редактирование:**
  - Текущий код `RolesList` загружает только права роли через `getPermissions`.
  - Необходимо:
    - добавить вспомогательный метод/сервис для чтения наследования (если в API есть `GET` для `role_inheritance`, либо использовать уже доступные данные, если бэкенд возвращает их в `Role`),
    - передавать `inheritsRoleId` в `RoleFormModal` для установки начального значения в селекте.

---

### 4. Визуализация унаследованных прав

- **Обновить `PermissionTree.vue`:**
  - Новый проп `inherited: string[]` (массив унаследованных `PermissionString`).
  - Для каждого `action.permissionString`:
    - если он есть в `inherited` и **нет** в локальном `modelValue`:
      - чекбокс рисуется в состоянии “выбрано и disabled” (роль не может снять унаследованное право),
      - можно добавить подсказку “унаследовано из [имя роли]” (имя роли можно передать отдельным пропом или пока ограничиться иконкой/стилем).
    - если он есть в `inherited` и в `modelValue`:
      - считать это “переопределением/дополнением” (поведение по договорённости; в MVP можно запретить дубль — не добавлять унаследованные права в локальный список).

---

### 5. Ограничения и валидация

- **Ограничить список родительских ролей:**
  - показывать только **кастомные** роли (не системные `OWNER/ADMIN/MEMBER/GUEST`), если такая логика согласована.
  - не позволять наследовать роль саму от себя.

- **Защита от циклов:**
  - На MVP-этапе:
    - фронт может просто запрещать выбирать в качестве родителя роли, которые уже явно помечены как наследники этой роли (если бэкенд возвращает граф наследования).
  - В идеале:
    - конечную проверку цикла должен делать бэкенд; фронт может показать сообщение об ошибке при 4xx.

---

### 6. Критерии готовности (Definition of Done для Этапа 1)

- В `RoleFormModal` есть селект “Наследование от роли”.
- Список для наследования показывает только допустимые роли (без самоссылающихся и, по договорённости, без системных).
- Унаследованные права визуально отличимы в `PermissionTree`.
- При сохранении роли создаётся/обновляется связь наследования через `roleService.addInheritance/removeInheritance`.
- При повторном открытии роли модалка корректно подставляет текущего родителя и отмечает унаследованные права.
