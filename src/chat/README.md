# Chat Module README

## Что это

`src/chat` — изолированный модуль виджета чата для ERP-контекста (текущий пользователь + текущий workspace), с возможностью:

- работать в компактном режиме (список/диалог поочередно),
- разворачиваться в расширенный режим (две колонки),
- использовать один и тот же стор и провайдеры данных независимо от UI.

Этот README фиксирует фактически проделанную работу и текущее состояние модуля вместо старых черновых описаний в `PLAN.md` / `TASK.md`.

---

## Что уже реализовано

## 1) Архитектура модуля

Сделано разделение на слои:

- `types.ts` — доменные типы (`ChatWidgetContext`, `ChatRecipient`, `ChatThread`, `ChatMessage`).
- `providers/*` — источники данных:
  - `backend-chat-provider.ts` (основной продовый провайдер через backend API + socket.io realtime),
  - `local-chat-provider.ts` (локальный/тестовый in-browser вариант),
  - `erp-chat-provider.ts` и базовые контракты `chat-provider.ts`, `chat-threads-provider.ts`.
- `model/use-chat-widget-store.ts` — единый Pinia store с бизнес-операциями.
- `ui/*` — слой интерфейса:
  - `ChatWidget.vue` (точка входа виджета),
  - `ui/widget/ChatRecipientsPanel.vue` (панель списка/поиска),
  - `ui/widget/ChatThreadPanel.vue` (панель сообщений),
  - сохранены более ранние `ui/slack/*` компоненты как альтернативный UI-слой.

Итог: UI не ходит напрямую в API, а работает через store.

## 2) Контекст ERP и изоляция по workspace

Реализовано:

- контекст берется либо из `props.context`, либо выводится из ERP-сторов (`user`, `workspace`) в `ChatWidget.vue`;
- store пересбрасывает данные при смене пользователя/воркспейса;
- загрузка получателей/чатов идет в рамках текущего workspace.

## 3) Основной пользовательский флоу

Поддерживается end-to-end:

- открыть виджет,
- увидеть список получателей,
- выбрать получателя,
- открыть/создать приватный тред,
- читать/отправлять сообщения,
- получать realtime-обновления.

Дополнительно:

- автофокус на поиск при открытии,
- автофокус на инпут сообщения при открытии треда,
- автопрокрутка вниз для новых сообщений (с проверкой расстояния до низа),
- заглушка, если тред не выбран.

## 4) Режимы отображения

Сделаны два состояния виджета:

- **Компактный**: список и диалог показываются по очереди.
- **Расширенный**: список слева и диалог справа одновременно.

Разница между режимами — только в layout, данные и store те же.

## 5) Анимации и плавность

Добавлено:

- плавное открытие/закрытие виджета (`chat-shell`),
- анимация переключения list/thread (`chat-view`),
- плавное изменение размеров при expand/collapse,
- auto-animate для списков через `@formkit/auto-animate`.

Для повторного использования добавлен helper:

- `shared/lib/use-auto-animate-el.ts`.

## 6) Иконки и UI-консистентность

Сделано:

- уход от прямого использования `@heroicons` в `ChatWidget` на локальные иконки из `shared/ui/icon`,
- добавлены локальные SVG-компоненты нужных иконок,
- в хедере чата используется общий `UserInfo` (тот же компонент, что и в списках) + статус online/offline.

## Обновления store (важные для UI)

В `use-chat-widget-store.ts` добавлены/используются:

- `openThread(thread)` — открыть уже существующий тред из списка,
- `closeThread()` — централизованно закрыть текущий тред и сбросить связанные данные,
- подписки на realtime события (`message.created`, `thread.updated`) с обновлением списка/сообщений.

Это позволило убрать прямые мутации store из UI-компонентов.

## Что отличается от исходных черновиков PLAN/TASK

Сравнение по сути:

- В `TASK.md` был пример на PocketBase SDK напрямую из фронта; в текущей реализации основной путь — через ваш backend API + socket.io (более совместимо с текущей архитектурой проекта).
- В `PLAN.md` заложена эволюция до PWA/интеграций; текущий код покрывает MVP-часть виджета и модульную изоляцию, но не полный вынос в отдельный runtime.
- Контракты и разделение слоев уже есть, но часть roadmap-фич (группы, full unread модель, богатые типы сообщений, внешние интеграции) пока не завершены.

## Текущее состояние по требованиям MVP

Готово:

- изолированный Vue-виджет чата,
- контекст от ERP,
- список получателей в рамках workspace,
- создание/открытие private чата,
- сообщения + realtime,
- расширенный и компактный режимы.

Частично/в процессе:

- online/offline пока зависит от доступности поля в данных получателя,
- визуальные детали и расположение контролов еще могут донастраиваться по UX-обратной связи,
- `ui/slack/*` и `ui/widget/*` сосуществуют; основной путь сейчас через `ui/widget/*`.

## Открытые задачи и рекомендации

1. **Унифицировать UI-слой**  
   Определить один “канонический” набор компонентов (`widget` или `slack`) и удалить дубль.

2. **Довести контракт статуса присутствия**  
   Явно добавить `online/isOnline` в типы/ответы backend и убрать эвристику.

3. **Закрыть вопрос с unread/cursor pagination**  
   В store уже есть заготовки, но нужен backend cursor API и полноценная модель непрочитанных.

4. **Докрутить групповые чаты**  
   Сейчас фокус на private MVP; для групп нужны title/avatar/participants и отдельные UX-сценарии.

5. **Почистить legacy-ветки**  
   После финального согласования UX удалить неиспользуемые компоненты/ветки верстки.
